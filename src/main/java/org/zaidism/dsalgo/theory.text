Time complexity is not the actual time taken to run an algorithm. Instead, it describes how the algorithm's running time grows relative to the input size. It's a theoretical measure of performance as input size increases.

Complexity Comparison: O(1) < O(log n) < O(n) < O(n log n) < O(k²)

O(1) – Constant time: execution time does not depend on input size.
O(log n) – Logarithmic time: grows slowly as input size increases (e.g., binary search).
O(n) – Linear time: time grows proportionally with input size.
O(n log n) – Log-linear time: typical for efficient sorting algorithms like mergesort or heapsort.
O(k²) – Quadratic time: often found in algorithms with nested loops (e.g., bubble sort), where k could be same as n.

Big O notation is a way to describe how fast an algorithm is or how much space it needs as the input gets bigger.

O(n) means the algorithm will take at most n steps (or use at most n space) when the input size is n. So, it won’t grow faster than n.
Omega Ω(n) means the algorithm will take at least n steps (or use at least n space) when the input size is n. So, it won’t be faster than that—it grows no slower than n.
Theta Θ(n) means the algorithm takes exactly about n steps (or space) as the input size grows. In other words, the algorithm’s performance grows at the same rate as the input size.

It’s like saying: The algorithm is no faster than n steps (like O(n)), and also no slower than n steps (like Ω(n)). So, it’s exactly proportional to n.

Quick comparison:
    O(n) = At most n steps (upper limit)
    Ω(n) = At least n steps (lower limit)
    Θ(n) = Exactly n steps (tight bound)

Space complexity: =  Auxiliary Space + Input space
 - Auxiliary Space: Space that take to solve the problem (Only considered)
 - Input space: Space that is needed to store input
